# 教学过程管理系统功能设计

## 1. 项目概述

### 1.1 项目背景

教学过程管理系统用于支撑学院教务、教学、考务、教材、教室、调课、毕业审核、教师考勤与教学日志等日常业务的信息化管理。本项目面向实习团队开发，按可落地、可演示、可扩展的原则进行范围收敛，不一次性完整复刻原需求文档中的全部外部系统集成。

本系统采用前后端双仓库开发。前端仓库使用 `Next.js + TypeScript`，负责页面、交互和接口调用；后端仓库使用 `NestJS + TypeScript + PostgreSQL + Prisma`，负责 REST API、业务规则、权限校验和数据持久化。两个仓库独立安装、运行、提交和发布，通过 Apifox 维护的接口文档、Mock 数据、联调环境和测试用例协作。

### 1.2 建设目标

- 实现教务与教学过程的核心业务线上化。
- 维护教师、学生、院系、专业、班级、课程、教室等基础数据。
- 支持教室申请、调课申请、考务安排、教材管理、毕业资格审核等流程闭环。
- 为教师、学生、教务人员、院系管理员、领导提供不同权限的工作台。
- 为后续接入统一身份认证、鹏达系统、指纹考勤系统、短信平台预留接口。

### 1.3 实习项目范围

本版本优先完成能够演示完整业务流程的核心功能：

- 必做模块：登录与权限、基础数据管理、教室申请、调课管理、考务管理、教学日志。
- 建议完成模块：教材管理、毕业资格审核、毕业生管理、专升本报名。
- 预留模块：CAS 单点登录、鹏达系统数据同步、指纹考勤同步、短信提醒、在线缴费。

## 2. 用户角色与权限

### 2.1 角色定义

| 角色 | 说明 | 核心权限 |
| --- | --- | --- |
| 系统管理员 | 维护系统配置和账号权限 | 用户管理、角色管理、基础配置、全量数据查看 |
| 教务处管理员 | 学校层面教务管理人员 | 基础数据维护、考务审批、调课审批、毕业审核 |
| 系部管理员 | 系部层面管理人员 | 本系部课程、教师、班级、调课审核、统计查看 |
| 教研室主任 | 教研室负责人 | 课程归属维护、教材审核、考核方式审核 |
| 教师 | 授课教师或课程负责人 | 课程资料上传、考核方式申报、调课申请、日志填写 |
| 学生 | 在校学生 | 考试查询、教室申请、重修申请、专升本报名 |
| 教材管理员 | 教材业务管理人员 | 教材征订、入库、发放、统计报表 |
| 领导 | 学院领导或分管领导 | 审批高课时调课、查看统计报表 |

### 2.2 权限原则

- 登录后根据角色生成菜单，禁止前端隐藏菜单替代后端鉴权。
- 后端所有 `/api` 接口必须校验登录状态和业务权限。
- 院系管理员只能管理本院系数据，系统管理员和教务处管理员可查看全校数据。
- 教师只能编辑自己负责的课程、资料、申请、日志。
- 学生只能查看或提交与本人相关的数据。

## 3. 技术架构

### 3.1 技术选型

| 层级 | 技术 | 说明 |
| --- | --- | --- |
| 前端框架 | Next.js | 使用 App Router，仅承载页面和前端交互 |
| 后端框架 | NestJS | 按业务模块提供 REST API、鉴权和业务服务 |
| 开发语言 | TypeScript | 两个仓库统一语言和字段命名规则 |
| UI 组件 | Ant Design 或 shadcn/ui | 后台系统优先选择表格、表单、弹窗成熟的组件库 |
| 状态管理 | React Query 或 SWR | 管理服务端数据缓存、刷新、错误态 |
| 数据库 | PostgreSQL | 适合结构化业务数据和统计查询 |
| ORM | Prisma | 统一 schema、迁移和类型生成 |
| 认证 | NestJS JWT + Cookie | 实习版先用账号密码登录，CAS 作为预留集成 |
| 接口协作 | Apifox | 维护接口文档、Mock 数据、环境变量和接口测试 |
| 测试 | Vitest、Playwright | 单元测试、接口测试和关键流程端到端测试 |

### 3.2 双仓库与分层设计

```text
Teaching-Management-System-fe/
  src/app/               Next.js 页面路由和布局
  src/components/        跨模块通用组件
  src/features/          按业务模块组织页面、表单、hooks
  src/lib/api/           统一 API 客户端
  src/types/             前端类型和 OpenAPI 生成类型

Teaching-Management-System-be/
  src/common/            公共 DTO、枚举、守卫、过滤器
  src/config/            环境变量和应用配置
  src/database/          Prisma 服务和数据访问基础设施
  src/modules/           controller、service、repository、dto
  prisma/                schema、migration、seed 数据
  test/                  API 集成测试
```

仓库地址与本地端口：

| 项目 | 仓库 | 默认地址 |
| --- | --- | --- |
| 前端 | `azc-pkk/Teaching-Management-System-fe` | `http://localhost:3000` |
| 后端 | `azc-pkk/Teaching-Management-System-be` | `http://localhost:3001/api` |

边界约定：

- 前端不得直接连接 PostgreSQL，也不得在 Next.js 中新增业务 API Route。
- 后端不得包含页面组件或依赖前端源码。
- 两个仓库不通过相对路径共享代码；公共接口类型以 Apifox/OpenAPI 文档为源生成。
- 前端通过 `NEXT_PUBLIC_API_BASE_URL` 切换 Mock、本地和测试环境。

| 外部系统 | 原需求 | 实习版处理方式 |
| --- | --- | --- |

| 鹏达系统 | 学期、课程、成绩、毕业班数据 | 预留导入接口，当前使用 seed 数据或 Excel/CSV 导入 |
| 指纹考勤系统 | 教师与学生考勤数据 | 预留同步接口，当前手动录入或 mock 同步 |

| 在线缴费 | 重修缴费 | 仅记录缴费状态，不接真实支付 |

## 4. 核心业务模块

### 4.1 教学基础数据管理

基础数据是其他模块的依赖，优先开发。

| 功能 | 说明 | 主要角色 |
| --- | --- | --- |
| 院系管理 | 维护院系、教研室、负责人 | 系统管理员、教务处管理员 |
| 专业管理 | 维护专业、所属院系、学制 | 教务处管理员 |
| 班级管理 | 维护班级、专业、年级、人数、辅导员 | 教务处管理员、系部管理员 |
| 教师管理 | 维护教师工号、姓名、类型、所属院系 | 系统管理员、系部管理员 |
| 学生管理 | 维护学号、姓名、班级、状态 | 系统管理员、系部管理员 |
| 课程管理 | 维护课程代码、名称、学分、类型、负责人 | 教务处管理员、教研室主任 |
| 教室管理 | 维护校区、楼栋、容量、类型、状态 | 教务处管理员 |
| 学期管理 | 设置当前学期、起止日期、教学周 | 教务处管理员 |

关键要求：

- 支持列表分页、关键字搜索、按院系/专业/学期筛选。
- 支持新增、编辑、软删除。
- 支持 CSV 或 Excel 导入导出作为加分项。
- 所有基础数据保留 `createdAt`、`updatedAt`、`createdBy`、`updatedBy` 审计字段。

### 4.2 考务管理

考务管理负责考核方式申报、试卷提交、考试安排、考试查询和统计。

| 功能 | 说明 | 主要角色 |
| --- | --- | --- |
| 考核方式申报 | 教师提交课程考核方式，教研室、系部、教务逐级审核 | 教师、教研室主任、系部管理员、教务处管理员 |
| 试卷提交管理 | 记录试卷提交状态、提交时间、审核结果 | 教师、系部管理员 |
| 期末考试安排 | 维护考试课程、班级、教室、时间、监考员 | 教务处管理员 |
| 补考安排 | 维护补考学生名单、课程、时间、教室 | 教务处管理员 |
| 重修考试申请 | 学生提交重修申请，教务审核，缴费状态预留 | 学生、教务处管理员 |
| 考试查询 | 学生和教师查看考试时间、地点、监考安排 | 学生、教师 |
| 考试统计 | 统计应考、实考、缺考人数 | 教务处管理员、领导 |

实习版不要求自动智能排考，先实现手动排考和冲突校验：

- 同一教室同一时间不能安排多场考试。
- 同一教师同一时间不能监考多场考试。
- 同一班级同一时间不能参加多场考试。

### 4.3 教材管理

教材管理覆盖征订、选取、审核、入库、发放和统计。

| 功能 | 说明 | 主要角色 |
| --- | --- | --- |
| 教材征订计划 | 按学期和课程维护教材需求 | 教材管理员、教师 |
| 教材选取审核 | 教师选择教材，教研室、系部、教务审核 | 教师、教研室主任、系部管理员、教务处管理员 |
| 自编教材管理 | 登记自编教材名称、作者、适用课程 | 教材管理员 |
| 新书入库 | 记录入库批次、数量、单价、供应商 | 教材管理员 |
| 教材发放 | 按班级或个人发放教材 | 教材管理员 |
| 教材统计 | 按学期、课程、班级、费用统计 | 教材管理员、领导 |

### 4.4 教室申请管理

教室申请面向全院师生和院内团体，兼顾电脑端与移动端。

| 功能 | 说明 | 主要角色 |
| --- | --- | --- |
| 教室查询 | 按校区、类型、容量、状态查询空闲教室 | 全体用户 |
| 教室申请 | 提交使用时间、用途、人数、申请说明 | 教师、学生、管理员 |
| 申请审批 | 管理员审核通过或驳回 | 教务处管理员 |
| 申请取消 | 申请人在开始前取消申请 | 申请人 |
| 冲突校验 | 同一教室同一时间不能重复占用 | 系统 |
| 使用统计 | 按教室、院系、用途统计申请情况 | 教务处管理员、领导 |

规则：

- 考务申请可提前两周申请多个教室并锁定。
- 普通申请最多提前一周。
- 同一申请人在同一时间段的申请数量可通过配置限制。
- 开关门能力只做字段预留，如 `accessWindowBeforeMinutes`、`accessWindowAfterMinutes`。

### 4.5 调课管理

调课管理用于教师发起调课申请，按课时数进入不同审批层级。

| 功能 | 说明 | 主要角色 |
| --- | --- | --- |
| 调课申请 | 教师填写课程、班级、原时间、新时间、教室、周次、原因 | 教师 |
| 系部审核 | 系部管理员审核本系部调课 | 系部管理员 |
| 教务审批 | 超过 8 学时进入教务处审批 | 教务处管理员 |
| 领导审批 | 超过 12 学时进入院领导审批 | 领导 |
| 补课反馈 | 教师填写补课完成情况 | 教师 |
| 调课统计 | 按系部、教师、课程统计调课量 | 系部管理员、教务处管理员、领导 |

状态流转：

```text
DRAFT -> SUBMITTED -> DEPT_APPROVED -> ACADEMIC_APPROVED -> LEADER_APPROVED -> COMPLETED
                      \-> REJECTED
```

审批规则：

- 小于等于 8 学时：系部审核通过后生效。
- 大于 8 且小于等于 12 学时：系部审核后进入教务审批。
- 大于 12 学时：系部、教务、领导三级审批。

### 4.6 毕业相关管理

毕业相关模块包含毕业资格审核、专升本报名、毕业生管理。

| 功能 | 说明 | 主要角色 |
| --- | --- | --- |
| 毕业资格审核 | 根据学分、课程通过情况、学籍状态生成审核结果 | 教务处管理员、系部管理员 |
| 专升本报名 | 学生提交报名，系统基于成绩和状态给出初审结果 | 学生、教务处管理员 |
| 毕业证编号 | 为毕业生维护毕业证编号 | 教务处管理员 |
| 上报库导出 | 导出毕业生上报数据 | 教务处管理员 |

实习版可简化为：

- 手动导入学生成绩或使用 seed 成绩数据。
- 毕业资格审核规则先配置为总学分是否达标、是否存在未通过必修课。
- 生成审核结果为 `通过`、`不通过`、`待人工复核`。

### 4.7 教师考勤与教学日志

| 功能 | 说明 | 主要角色 |
| --- | --- | --- |
| 教师考勤记录 | 记录教师上课签到、签退、异常说明 | 教师、系部管理员 |
| 教学日志填写 | 教师课后填写授课内容、作业、学生出勤情况 | 教师 |
| 日志提交限制 | 默认要求每周末前提交，可做提醒 | 系统 |
| 学生周报表 | 记录班级或学生维度周报信息 | 学生、教师 |
| 日志审核与统计 | 查看教师日志提交率、异常考勤 | 系部管理员、教务处管理员 |

## 6. 后端设计

### 6.1 API 设计原则

- 使用 RESTful 风格。
- 所有接口返回统一 `ApiResponse<T>`。
- 列表接口统一支持分页。
- 创建、修改、审批接口必须记录操作人和操作时间。
- 删除优先使用软删除。
- 重要业务状态变更写入审批记录或操作日志。
- 所有接口必须先登记到 Apifox，包含请求参数、响应示例、错误示例和 Mock 规则。

### 6.2 统一响应类型

```ts
export type ApiResponse<T> = {
  success: boolean;
  data?: T;
  error?: {
    code: string;
    message: string;
    details?: unknown;
  };
};

export type PageResult<T> = {
  items: T[];
  page: number;
  pageSize: number;
  total: number;
};

export type PageQuery = {
  page?: number;
  pageSize?: number;
  keyword?: string;
};
```

这些类型的字段定义以 Apifox/OpenAPI 为准。后端维护 DTO 和接口文档，前端从 OpenAPI 生成请求响应类型到 `src/types/generated`；禁止通过复制后端源码维持类型一致性。

### 6.3 枚举类型

```ts
export type UserRole =
  | "SYSTEM_ADMIN"
  | "ACADEMIC_ADMIN"
  | "DEPARTMENT_ADMIN"
  | "TEACHING_RESEARCH_DIRECTOR"
  | "TEACHER"
  | "STUDENT"
  | "TEXTBOOK_ADMIN"
  | "LEADER";

export type ApprovalStatus =
  | "DRAFT"
  | "SUBMITTED"
  | "APPROVED"
  | "REJECTED"
  | "CANCELED"
  | "COMPLETED";

export type Semester = {
  id: string;

  startDate: string;
  endDate: string;
  isCurrent: boolean;
};
```

| 方法 | 路由 | 说明 |

| `DELETE` | `/api/base/teachers/:id` | 删除教师 |
| `GET` | `/api/base/students` | 学生列表 |
| `GET` | `/api/base/courses` | 课程列表 |
| `GET` | `/api/base/classrooms` | 教室列表 |

### 6.5 Apifox 使用规范

Apifox 作为前后端接口协作的统一入口，接口字段以 Apifox 文档、TypeScript 类型和后端实际返回三者一致为准。

| 使用场景 | 负责人 | 要求 |
| --- | --- | --- |
| 接口目录 | 后端工程师 | 按基础数据、考务、教材、教室申请、调课、毕业、教学日志分组 |
| 公共模型 | 后端工程师 | 维护 `ApiResponse<T>`、`PageResult<T>`、错误响应、登录用户等公共结构 |
| Mock 数据 | 前端工程师、后端工程师 | 前端开发前补充可用 Mock，字段必须贴近真实业务 |
| 环境管理 | 后端工程师 | 至少配置 Mock、本地开发、测试环境三个环境 |
| 接口联调 | 前端工程师、后端工程师 | 后端完成后用 Apifox 验证状态码、响应结构和权限错误 |
| 接口测试 | 后端工程师 | 对新增、编辑、审批、删除、越权访问补充测试用例 |

推荐目录：

```text
教学过程管理系统
  01 登录与权限
  02 基础数据
  03 教室申请
  04 调课管理
  05 考务管理
  06 教材管理
  07 毕业相关
  08 教师考勤与教学日志
  09 系统配置
```

接口变更流程：

1. 先在 Apifox 修改接口文档、请求参数、响应示例和 Mock。
2. 后端按接口文档实现或调整 API。
3. 前端根据 Apifox Mock 或真实环境完成页面调用。
4. 联调通过后，在 Apifox 中保存测试用例。
5. 涉及字段变更时，后端同步更新 DTO 和 Prisma schema，前端重新生成接口类型。

## 9. 前后端协作约定

- 前后端分别在各自仓库建立功能分支和 Pull Request，不把两个项目放回同一个 Git 仓库。
- 后端先提供 Prisma schema、seed 数据，并在 Apifox 中维护 OpenAPI 接口清单。
- 前端可先使用 Apifox Mock API 开发页面，接口稳定后切换真实 API。
- 每个模块至少包含列表页、详情或编辑页、状态展示、错误提示。
- 接口字段以 Apifox 文档、OpenAPI 生成类型和后端返回为准，禁止在两个仓库手工维护不同字段名。
- 审批类接口必须返回最新状态和审批记录，方便前端刷新时间线。
- 接口发生新增、删除、字段变更、状态码变更时，必须先更新 Apifox 并通知对应前端或后端同学。
- 两个仓库的代码提交前都必须通过 `typecheck`、`lint`、关键测试和 `build`。

## 10. 验收清单

### 10.1 功能验收

- 不同角色登录后只能访问授权菜单和接口。
- 教师、学生、课程、教室、班级基础数据可增删改查、筛选、分页。
- 教室申请能校验时间冲突、提前申请限制，并支持审批和取消。
- 调课申请能根据课时数进入不同审批层级。
- 考务安排能关联课程、班级、教室、监考教师，并校验冲突。
- 教材征订、入库、发放至少完成可演示闭环。
- 毕业资格审核能根据规则生成审核结果。
- 教师能填写教学日志，管理员能查看提交统计。

### 10.2 工程验收

- 数据库可通过 Prisma migration 初始化。
- seed 数据能生成演示账号和基础业务数据。
- API 有统一响应结构和统一错误处理。
- 前端表单有必填、格式、时间范围校验。
- 关键审批流程有集成测试。
- 权限和越权访问有测试。
- 项目 README 包含启动、环境变量、数据库初始化说明。

## 12. 开发默认假设

- 系统第一版以实习演示和小组协作为目标，优先保证核心流程闭环。
- CAS、鹏达系统、指纹考勤系统、短信平台、在线缴费均作为后续真实项目集成点。
- 外部系统数据在第一版使用 seed、mock、CSV 或 Excel 导入模拟。
- 前后端分别维护独立 TypeScript 工程，接口类型通过 Apifox/OpenAPI 生成并校验。
- 数据库默认使用 PostgreSQL，开发环境可通过 Docker 启动。
- UI 以后台管理系统为主，强调清晰、高效、稳定，不做营销型页面。
